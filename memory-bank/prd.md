# PRD：宫颈阴道镜病例检索 RAG（V2 重构阶段）

## 1. 阅读导航（先看这里）
1. 先读 `architecture.md`：理解系统全貌与脚本职责。
2. 再读 `implementation-plan.md`：理解 step 门禁与执行顺序。
3. 最后读 `progress.md`：确认当前状态、提交与阻塞信息。

## 2. 背景与问题
V1 已完成可运行基线，但存在三个核心问题：
1. 检索链路手写比例高，重复造轮子，可读性与协作性不足。
2. 架构与进度文档表达不够直观，人类与 AI 上下文切换成本高。
3. 需要兼顾“代码简洁”与“关键边界防护”，避免两个极端。

## 3. V2 目标
1. 采用 LangChain 渐进迁移，优先替换检索编排链路。
2. 文档可读性升级：图内中文注释 + 表格化关键状态。
3. 保留分层错误体系，确保定位问题时仍可精确归因。
4. 让 AI 与人类都能快速判断“当前该做什么、下一步是什么”。

## 4. 范围与非目标
### 4.1 当前范围（STEP-11）
1. 仅改 `memory-bank` 五个必读文件，不改 `src/` 与 `tests/` 行为。
2. `architecture.md`：图节点必须带中文职责注释。
3. `progress.md`：使用 `提交时间` 列替代 `更新时间`。
4. 五文档术语统一，减少执行歧义。

### 4.2 非目标（本轮）
1. 不改变 `HSIL | LSIL | Uncertain` 判定逻辑与阈值。
2. 不新增图像向量检索或服务化能力。
3. 不进行一次性全面代码重写。

## 5. 关键术语（文档协作口径）
1. `AI ReviewCode`：由 AI 对本轮交付物做一致性与质量复核。
2. `提交时间`：仅在 `已提交=✓` 时填写 commit 时间；未提交统一 `-`。
3. `待提交`：功能/文档已完成，但尚未 commit。
4. `blocked`：存在阻塞，必须提供解除条件。

## 6. 约束与兼容策略
1. 外部 CLI 输出契约保持兼容：
   - `similar_cases`
   - `tendency`
   - `reason`
   - `disclaimer`
   - `meta`
2. 错误分层策略保留，不退化为模糊异常。
3. Step 执行遵守门禁：未收到 `start next step` 不进入下一步。

## 7. 文档阶段验收标准（STEP-11）
1. `architecture.md` 至少 3 张图（当前架构图、时序图、目标架构图）。
2. 三张图中每个脚本节点均包含中文职责注释。
3. `progress.md` 使用 `提交时间` 列，且未提交步骤统一 `-`。
4. `implementation-plan.md` 与 `progress.md` 的状态/门禁口径一致。
5. `tech-stack.md` 的“现状/目标/保留项”与架构图一致。

## 8. 风险与应对
1. 风险：图注过长导致图拥挤。
   - 应对：图内职责注释保持短句，细节写表格。
2. 风险：提交字段填报不一致。
   - 应对：在 `progress.md` 固化提交时间填写规则。
3. 风险：文档口径漂移影响执行。
   - 应对：每轮先读五文件并做 AI ReviewCode。

## 9. 当前结论（2026-02-23）
1. 本轮按需求变化/重构执行。
2. 当前聚焦 STEP-11 文档修订收口，不进入 STEP-12 开发。
3. only executing current step scope.
